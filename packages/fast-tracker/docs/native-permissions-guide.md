# Native App Permissions Guide

This guide explains how to configure and request permissions for the Time Tracker app when packaged as a native application for Android and iOS.

## Required Permissions

The Time Tracker app currently requires minimal permissions:

- **Internet access**: Required for online functionality
- **Storage access**: Required for IndexedDB to work properly

## Android Permissions

### Manifest Permissions

When packaging the PWA for Android, the following permissions should be included in the Android manifest:

```xml
<!-- Required for internet access -->
<uses-permission android:name="android.permission.INTERNET" />

<!-- Required for PWA to work offline -->
<uses-permission android:name="android.permission.ACCESS_NETWORK_STATE" />
```

### Configuring Permissions with PWA Builder

When using PWA Builder for Android:

1. During the packaging process, you'll be asked to configure permissions
2. Ensure the INTERNET permission is enabled
3. You can leave other permissions disabled unless you plan to add features that require them

### Configuring Permissions with Bubblewrap

If using Bubblewrap CLI:

1. The `manifest.json` file generated by Bubblewrap will include basic permissions
2. Edit the `android.manifestPlaceholders` section to add or remove permissions
3. Rebuild the APK after making changes

## iOS Permissions

iOS handles permissions differently from Android. Most permissions are requested at runtime rather than being declared in the app manifest.

### Basic Permissions

For the current Time Tracker functionality, you don't need to request special permissions on iOS beyond what's automatically granted to web applications.

### Configuring with PWA Builder

When using PWA Builder for iOS:

1. The generated Xcode project will include basic entitlements
2. No additional configuration is needed for the current feature set

## Future Permission Considerations

If you plan to add these features in the future, you'll need additional permissions:

### Notifications

**Android:**
```xml
<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
```

**iOS:**
Add to Info.plist:
```xml
<key>NSUserNotificationUsageDescription</key>
<string>We need to send you notifications about your time tracking activities.</string>
```

### Calendar Integration

**Android:**
```xml
<uses-permission android:name="android.permission.READ_CALENDAR" />
<uses-permission android:name="android.permission.WRITE_CALENDAR" />
```

**iOS:**
Add to Info.plist:
```xml
<key>NSCalendarsUsageDescription</key>
<string>We need access to your calendar to add time tracking entries.</string>
```

## Requesting Permissions at Runtime

For permissions that require user consent:

### Android (API Level 23+)

```java
// Check if permission is granted
if (ContextCompat.checkSelfPermission(context, Manifest.permission.PERMISSION_NAME)
        != PackageManager.PERMISSION_GRANTED) {
    // Request permission
    ActivityCompat.requestPermissions(activity,
            new String[]{Manifest.permission.PERMISSION_NAME},
            REQUEST_CODE);
}
```

### iOS

```swift
// Request notification permission
UNUserNotificationCenter.current().requestAuthorization(options: [.alert, .sound, .badge]) { granted, error in
    // Handle response
}
```

## Best Practices

1. **Request permissions only when needed**: Don't request permissions at app startup if they're not immediately needed
2. **Explain why**: Always explain to users why your app needs a permission before requesting it
3. **Handle denials gracefully**: Provide alternative functionality if a permission is denied
4. **Provide a way to change permissions**: Include instructions on how to update permissions in device settings

## Testing Permissions

Before releasing your app:

1. Test on real devices (not just emulators)
2. Test the permission request flow
3. Test what happens when permissions are denied
4. Test what happens when permissions are revoked after being granted
